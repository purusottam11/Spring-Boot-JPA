Part I — Foundations

Three names, three different things. JPA is a specification (just interfaces + annotations, zero code that runs). Hibernate is the implementation that actually generates SQL and talks JDBC. Spring Data JPA sits on top and writes your DAO layer for you — you declare a repository interface, Spring builds a proxy implementing it at startup.

JPA is the interface, Hibernate is the class that implements it, and Spring Data JPA is like Lombok for DAOs — it generates the boilerplate you'd otherwise hand-write.
One request: JSON in → DTO → entity → SQL → rows. The pulse traces the path every request takes.
📸 On Instagram

When you tap ❤️ on a post: the client calls POST /api/posts/42/likes → controller validates → service opens a transaction, loads the Post, inserts a Like row → Hibernate flushes an INSERT through a pooled JDBC connection → InnoDB writes the row and the unique index (user_id, post_id) rejects double-likes.

In interviews, nail this one-liner: "Spring Data JPA is not an ORM. Hibernate is the ORM; Spring Data JPA is a repository abstraction over JPA's EntityManager."

Two dependencies get you everything: the JPA starter (pulls Hibernate + Spring Data + HikariCP) and the MySQL driver. Configuration lives in application.yml.

📸 On Instagram

You'd never let Hibernate update a table holding 2 billion posts — an accidental column rename becomes a "missing column" + orphaned data. Schema changes ship as versioned Flyway scripts (V7__add_posts_location.sql), and ddl-auto: validate guarantees entities and schema agree at boot.

open-in-view: true (the default!) keeps the DB session open through view rendering, hiding lazy-loading bugs and holding connections longer. Turn it off on day one — chapters 20 and 27 explain the failures it masks.

📸 The cast

User owns one Profile (1:1), publishes many Posts (1:N), each Post has many Comments (1:N). A user likes many posts and a post is liked by many users — Like is that N:M with a timestamp. Followers is the interesting one: an N:M from users to users (self-referencing).

Part II — Mapping

An entity is a Java class Hibernate maps to a table: @Entity marks it, @Id marks the primary key, and everything else customizes column names, types, and constraints.

📸 On Instagram

The User entity: id, unique username/email, an account status enum, and a signup timestamp. This single class shows 90% of the annotations you'll use daily.

User.java
@Entity
@Table(name = "users")                       // "user" is reserved-ish in SQL; be explicit
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)   // MySQL AUTO_INCREMENT
    private Long id;

    @Column(nullable = false, unique = true, length = 30)
    private String username;

    @Column(nullable = false, unique = true)
    private String email;

    @Enumerated(EnumType.STRING)             // stores "ACTIVE", not 0
    @Column(nullable = false, length = 20)
    private AccountStatus status = AccountStatus.ACTIVE;

    @Lob                                     // TEXT / LONGTEXT column
    private String pushToken;

    @Transient                               // never persisted
    private int unreadBadgeCount;

    @Column(name = "created_at", nullable = false, updatable = false)
    private Instant createdAt = Instant.now();

    protected User() {}                      // JPA needs a no-arg constructor
    // getters/setters…
}

ID generation on MySQL

StrategyHowMySQL verdict
IDENTITYAUTO_INCREMENT; DB assigns id on INSERTThe standard choice
SEQUENCEDB sequence objectMySQL has no sequences — Hibernate falls back to a table
AUTOHibernate picksMay silently create a *_seq table; be explicit instead
UUIDApp-generated UUIDFine, but random UUIDs fragment InnoDB's clustered index
EnumType.ORDINAL (the default!) stores the enum's position. Insert SUSPENDED before ACTIVE in the enum later and every stored user silently changes status. Always EnumType.STRING.
IDENTITY has one cost: Hibernate must execute the INSERT immediately to learn the id, which disables JDBC insert batching (chapter 26). For MySQL you usually accept this trade.

Each user has exactly one profile. The cleanest mapping is shared primary key with @MapsId: the profile's PK is the user's PK — one column doing PK + FK duty, no extra index.

📸 On Instagram

Why split at all? users is read on every request (auth, mentions, likes); bio/avatar/website only matter on the profile screen. Splitting keeps the hot table narrow.

Profile.java — shared-PK mapping
@Entity
@Table(name = "profiles")
public class Profile {

    @Id
    private Long id;                          // same value as user.id

    @OneToOne(fetch = FetchType.LAZY)
    @MapsId                                   // reuse the User's id as my PK
    @JoinColumn(name = "user_id")
    private User user;

    private String bio;
    private String avatarUrl;
    private String website;
}

// creating both together
User u = new User("purusottam", "p@x.dev");
Profile p = new Profile();
p.setUser(u);                                 // id copied from user at persist
userRepository.save(u);
profileRepository.save(p);

The alternative — a plain FK column with @JoinColumn(name = "user_id", unique = true) on Profile — also works; you then add @OneToOne(mappedBy = "user") on User only if you truly need user.getProfile().

The mappedBy (parent) side of a one-to-one cannot be truly LAZY. Hibernate must query the child table anyway to decide between proxy and null. If you see a surprise SELECT … FROM profiles on every user load, this is why. Fix: query from the owning side, or use a dedicated repository call / projection.

The workhorse relationship. The many side (Post.user) owns the FK column and is what you should always map. The one side (User.posts) is optional sugar — add it only when navigating that direction genuinely helps.

Post.java + Comment.java
@Entity @Table(name = "posts")
public class Post {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)   // owning side → FK user_id
    @JoinColumn(name = "user_id")
    private User user;

    private String caption;
    private String imageUrl;
    private Instant createdAt = Instant.now();

    @OneToMany(mappedBy = "post", cascade = CascadeType.ALL,
               orphanRemoval = true)                        // inverse side
    private List<Comment> comments = new ArrayList<>();

    public void addComment(Comment c) {      // keep both sides in sync
        comments.add(c);
        c.setPost(this);
    }
}

@Entity @Table(name = "comments")
public class Comment {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "post_id")
    private Post post;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "user_id")
    private User author;

    private String body;
}
📸 On Instagram

Post → Comments is bidirectional because a post screen always shows its comments. But User.posts? A celebrity has 10,000 posts — you'd never call user.getPosts() and load them all. Instead you query: postRepository.findByUserId(id, pageable). Unbounded collections should be queries, not fields.

A @OneToMany without mappedBy creates a hidden join table (or forces extra UPDATEs). If you map both directions, mappedBy on the collection side is mandatory — it tells Hibernate "the FK lives over there."
Always add the sync helper (addComment). Setting only one side is the #1 cause of "why is my FK null" bugs in bidirectional mappings.

Two flavors here. Followers: a self-referencing @ManyToMany from User to User. Likes: starts as N:M, but the moment you need created_at on the relationship, you must promote the join table to a real entity.

User.java — self-referencing followers
@ManyToMany
@JoinTable(name = "followers",
    joinColumns        = @JoinColumn(name = "follower_id"),   // me
    inverseJoinColumns = @JoinColumn(name = "following_id"))  // who I follow

@ManyToMany(mappedBy = "following")

📸 Why Like is an entity, not @ManyToMany

Instagram shows "liked 2h ago" and orders like-notifications by time. A raw @ManyToMany join table can hold only the two FKs — no timestamp, no id, no extra state. Rule of thumb: the moment the relationship itself has data, it becomes an entity.

Like.java — join table promoted to entity
@Entity
@Table(name = "likes",
       uniqueConstraints = @UniqueConstraint(
           name = "uq_like_user_post",
           columnNames = {"user_id", "post_id"}))     // one like per user per post
public class Like {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "user_id")
    private User user;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "post_id")
    private Post post;

    private Instant createdAt = Instant.now();
}
Never map likes as Post.likedBy : Set<User> in a real system — a viral post has 5M likes and one getLikedBy() call OOMs the JVM. Count with likeRepository.countByPostId(id), check with existsByUserIdAndPostId(…).
For celebrity-scale followers, the same logic applies: keep the @ManyToMany for small curated sets (e.g., "close friends", capped at 100s) and use a Follow entity + paged queries for the open-ended graph.
Part III — Repositories & Queries

Declare an interface, extend JpaRepository<Entity, IdType>, done — Spring generates a proxy with ~20 CRUD methods at startup. You never write an implementation class.

The hierarchy: Repository (marker) → CrudRepository (save, findById, delete) → PagingAndSortingRepository (+ paging) → JpaRepository (+ flush, saveAll, batch deletes, JPA-specific extras).

PostRepository.java + service usage
public interface PostRepository extends JpaRepository<Post, Long> { }

@Service
@RequiredArgsConstructor
public class PostService {
    private final PostRepository posts;

    @Transactional
    public Post publish(Long userId, String caption, String imageUrl) {
        User author = userRepository.getReferenceById(userId); // proxy, no SELECT
        return posts.save(new Post(author, caption, imageUrl)); // INSERT
    }

    @Transactional(readOnly = true)
    public Post get(Long id) {
        return posts.findById(id)                               // Optional<Post>
                    .orElseThrow(() -> new PostNotFoundException(id));
    }

    @Transactional
    public void delete(Long id) {
        posts.deleteById(id);                                   // SELECT + DELETE
    }
}
📸 On Instagram

Publishing a post needs the author only as an FK value. getReferenceById returns a lazy proxy holding just the id — no SELECT … FROM users. findById would work too, but at feed-write volume that's millions of pointless reads/day.

save() is really persist-or-merge: if the id is null it INSERTs; otherwise it merges (may SELECT first). And inside a transaction you often don't need save() at all for updates — dirty checking (ch. 19) writes changes automatically.

Spring parses the method name and generates the query. findByUsername becomes WHERE username = ? — zero SQL written, checked at startup (typos fail fast at boot, not in prod).

JPQL looks like SQL but targets entities and their fields, not tables and columns. Hibernate translates it to MySQL dialect. You join through relationships (p.user), never through FK columns.

📸 The home feed query

"Show me recent posts from everyone I follow" — the single most important query in the app. It joins through the follow relationship in one statement.

PostRepository.java
public interface PostRepository extends JpaRepository<Post, Long> {

    // feed: posts authored by users that :viewer follows
    @Query("""
        SELECT p FROM Post p
        WHERE p.user IN (SELECT f.following FROM Follow f
                         WHERE f.follower.id = :viewerId)
        ORDER BY p.createdAt DESC
        """)
    Slice<Post> findFeed(@Param("viewerId") Long viewerId, Pageable pageable);

    // bulk write: JPQL UPDATE bypasses entities entirely
    @Modifying(clearAutomatically = true)
    @Query("UPDATE Post p SET p.archived = true WHERE p.user.id = :userId")
    int archiveAllByUser(@Param("userId") Long userId);
}
  • Named params (:viewerId) over positional ?1 — refactor-safe.
  • @Modifying is required for UPDATE/DELETE. These skip the persistence context, so clearAutomatically = true evicts now-stale managed entities.
  • JPQL is portable across databases; it can't use MySQL-only features — that's chapter 11's job.
Bulk @Modifying queries do not fire entity lifecycle events, @Version checks, or cascades. If an already-loaded Post sits in the persistence context, it still shows archived = false after the bulk update — hence the clear.

Same @Query, add nativeQuery = true, and you're writing raw MySQL against real tables. Reach for it when you need something JPQL can't express: window functions, index hints, ON DUPLICATE KEY, JSON functions.

📸 Trending: top 3 posts per user this week

"Top-N per group" needs ROW_NUMBER() OVER (PARTITION BY …) — a window function JPQL doesn't have. Classic native-query territory.

PostRepository.java
@Query(value = """
    SELECT * FROM (
        SELECT p.*,
               ROW_NUMBER() OVER (PARTITION BY p.user_id
                                  ORDER BY p.like_count DESC) AS rn
        FROM posts p
        WHERE p.created_at >= NOW() - INTERVAL 7 DAY
    ) ranked
    WHERE ranked.rn <= 3
    """,
    nativeQuery = true)
List<Post> findWeeklyTop3PerUser();

// pagination with native SQL needs an explicit count query
@Query(value = "SELECT * FROM posts WHERE user_id = :uid ORDER BY created_at DESC",
       countQuery = "SELECT count(*) FROM posts WHERE user_id = :uid",
       nativeQuery = true)
Page<Post> pageByUser(@Param("uid") Long userId, Pageable pageable);
Native queries trade away portability, startup validation, and automatic Sort mapping (property names ≠ column names). Keep them rare, name them clearly, and cover each with a test — a typo here surfaces at runtime, not boot.
Returning entities from native SQL works if the SELECT provides all mapped columns. For partial columns, return an interface projection (ch. 24) instead of wrestling with @SqlResultSetMapping.

Queries built as Java objects instead of strings — for when the query's shape isn't known until runtime. Powerful, verbose, and mostly hidden behind Specifications (ch. 13) in modern code. Know it because interviewers ask, and because Specifications compile down to it.

📸 Advanced post search

The Explore filter screen: caption text is optional, author is optional, date range is optional. String-concatenating JPQL for 2³ combinations is a bug factory; Criteria builds exactly the WHERE clause the request needs.

PostSearchDao.java
public List<Post> search(String caption, Long userId, Instant since) {
    CriteriaBuilder cb = em.getCriteriaBuilder();
    CriteriaQuery<Post> cq = cb.createQuery(Post.class);
    Root<Post> post = cq.from(Post.class);

    List<Predicate> where = new ArrayList<>();
    if (caption != null)
        where.add(cb.like(cb.lower(post.get("caption")),
                          "%" + caption.toLowerCase() + "%"));
    if (userId != null)
        where.add(cb.equal(post.get("user").get("id"), userId));
    if (since != null)
        where.add(cb.greaterThanOrEqualTo(post.get("createdAt"), since));

    cq.select(post)
      .where(where.toArray(new Predicate[0]))
      .orderBy(cb.desc(post.get("createdAt")));

    return em.createQuery(cq).setMaxResults(50).getResultList();
}
String paths like post.get("caption") aren't compile-checked — rename the field and this breaks at runtime. The JPA static metamodel (Post_.caption, generated at build time) fixes that, at the cost of an annotation-processor setup.

Spring Data's ergonomic wrapper over Criteria: each filter becomes a small, named, reusable Specification, and you chain them with .and() / .or() like Lego bricks.

PostSpecs.java + usage
public interface PostRepository
        extends JpaRepository<Post, Long>, JpaSpecificationExecutor<Post> { }

public final class PostSpecs {
    public static Specification<Post> captionContains(String q) {
        return (root, cq, cb) ->
            cb.like(cb.lower(root.get("caption")), "%" + q.toLowerCase() + "%");
    }
    public static Specification<Post> byAuthor(Long userId) {
        return (root, cq, cb) -> cb.equal(root.get("user").get("id"), userId);
    }
    public static Specification<Post> postedAfter(Instant t) {
        return (root, cq, cb) -> cb.greaterThanOrEqualTo(root.get("createdAt"), t);
    }
}

// service — assemble only the filters the request actually sent
Specification<Post> spec = Specification.allOf(
    q      != null ? PostSpecs.captionContains(q)  : null,
    userId != null ? PostSpecs.byAuthor(userId)    : null,
    since  != null ? PostSpecs.postedAfter(since)  : null);

Page<Post> result = postRepository.findAll(spec, pageable);   // paging for free
📸 On Instagram

Same Explore search as chapter 12, but each rule now has a name, a home, and a unit test — and moderation tooling reuses byAuthor() without copy-pasting predicates.

Specifications + findAll(spec, pageable) is the cleanest answer to "build me a search endpoint with N optional filters" — a very common interview design question.

Pass a Pageable into any repository method and Spring adds LIMIT/OFFSET + ORDER BY. The key decision is the return type: Page runs an extra COUNT query; Slice just checks "is there one more?"

📸 Infinite scroll = Slice, not Page

The feed never shows "page 3 of 41,209" — it only needs hasNext. Skipping COUNT(*) over a hundred-million-row posts table on every scroll is a real win. Use Page where you render numbered pagination (admin dashboards).

Controller + repository
// GET /api/users/7/posts?page=0&size=12&sort=createdAt,desc
@GetMapping("/api/users/{id}/posts")
public Slice<PostCard> grid(@PathVariable Long id,
                            @PageableDefault(size = 12, sort = "createdAt",
                                             direction = Sort.Direction.DESC)
                            Pageable pageable) {
    return postRepository.findByUserId(id, pageable);
}

// repository
Slice<PostCard> findByUserId(Long userId, Pageable pageable);

// building one manually
Pageable p = PageRequest.of(0, 12,
        Sort.by("createdAt").descending().and(Sort.by("id").descending()));
Deep OFFSET is a scan: LIMIT 12 OFFSET 120000 reads and throws away 120k rows. For endless feeds, switch to keyset (seek) pagination: WHERE created_at < :lastSeen ORDER BY created_at DESC LIMIT 12 — constant-time regardless of depth, and it rides the (user_id, created_at) index from chapter 23.
Always add a unique tiebreaker (id) to your sort. Two posts created in the same millisecond can otherwise swap between pages, and users see duplicates while scrolling.
Part IV — Transactions & the Persistence Context

@Transactional wraps a method in begin → commit/rollback via an AOP proxy. Put it on service methods — that's your unit-of-work boundary. Default rollback rule: unchecked exceptions roll back, checked exceptions do not.

📸 Publishing a post is two writes or none

Insert the post and bump profile.postCount. If the counter update fails, the post insert must vanish too — otherwise profile says "141 posts" while the grid shows 142.

PostService.java
@Service
public class PostService {

    @Transactional                                  // one atomic unit
    public Post publish(Long userId, CreatePostRequest req) {
        User author = users.getReferenceById(userId);
        Post post = posts.save(new Post(author, req.caption(), req.imageUrl()));

        Profile profile = profiles.findById(userId).orElseThrow();
        profile.incrementPostCount();               // dirty checking → UPDATE
        return post;                                // commit flushes both writes
    }

    @Transactional(readOnly = true)                 // hint: no flush, driver optimizations
    public PostDetail getDetail(Long postId) { … }

    @Transactional(rollbackFor = ReceiptException.class)   // checked → opt-in
    public void purchasePromotion(Long postId) throws ReceiptException { … }
}
  • Rollback: RuntimeException/Error → automatic. Checked exceptions → commit anyway unless rollbackFor says otherwise. This surprises everyone once.
  • readOnly = true: skips dirty-check snapshots and flush — measurable on hot read paths like the feed.
  • Where it fails silently: @Transactional on private methods, or this.otherMethod() self-invocation — the proxy is bypassed, no transaction starts.
Catching an exception inside the transactional method and swallowing it doesn't undo the rollback mark. If a RuntimeException already crossed a transactional boundary, commit throws UnexpectedRollbackException. Handle by letting it propagate, or catch outside the boundary.

Propagation answers one question: when a transactional method calls another, does the inner one join the existing transaction, suspend it and start its own, or something else?

📸 Moderation audit must outlive the rollback

When auto-moderation rejects a post, the publish transaction rolls back — but compliance requires the "content rejected" audit row to persist. REQUIRES_NEW gives the audit write its own committed transaction.

AuditService.java
@Service
public class AuditService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void record(String event, Long userId) {
        auditRepository.save(new AuditEntry(event, userId, Instant.now()));
    }   // commits HERE, independent of the caller's fate
}

@Service
public class PostService {
    @Transactional
    public Post publish(Long userId, CreatePostRequest req) {
        Post post = posts.save(new Post(…));
        if (moderation.isBlocked(post)) {
            audit.record("POST_REJECTED", userId);   // separate tx, separate connection
            throw new ContentPolicyException();       // outer tx rolls back; audit stays
        }
        return post;
    }
}
Two gotchas: (1) REQUIRES_NEW borrows a second connection while the first is suspended — under load this can deadlock an undersized pool. (2) Self-invocation strikes again: calling this.record(…) from the same class ignores the annotation. It must cross a bean proxy.

Isolation decides what one running transaction can see of others. Stronger isolation = fewer anomalies = less concurrency. MySQL/InnoDB defaults to REPEATABLE_READ; most other RDBMSs default to READ_COMMITTED.

LevelDirty readNon-repeatable readPhantom read

READ_COMMITTEDpreventedpossiblepossible
REPEATABLE_READ (MySQL default)preventedpreventedmostly prevented*
SERIALIZABLEpreventedpreventedprevented

* InnoDB's MVCC snapshots + gap locks block most phantoms at REPEATABLE_READ — a favorite interview nuance.

📸 Anomalies in Instagram terms

Dirty read: analytics counts a like from a transaction that then rolls back — a like that never existed. Non-repeatable read: a report reads follower_count twice in one tx and gets 999 then 1000. Phantom: "posts in the last hour" returns 5 rows, re-run in the same tx returns 6.

Setting it per method
// consistent multi-read snapshot for a payout/analytics job
@Transactional(isolation = Isolation.REPEATABLE_READ, readOnly = true)
public CreatorStats computeMonthlyStats(Long userId) {
    long posts    = postRepository.countByUserId(userId);
    long likes    = likeRepository.countByPostUserId(userId);
    long follows  = followRepository.countByFollowingId(userId);
    return new CreatorStats(posts, likes, follows);   // one consistent snapshot
}
Don't reach for SERIALIZABLE to fix a lost-update on like counters — that's a locking problem (ch. 22) or an atomic-update problem (UPDATE … SET like_count = like_count + 1), not an isolation problem.

The persistence context is Hibernate's per-transaction working memory: a map of id → managed entity. Every entity is in exactly one of four states, and knowing which explains 80% of "why didn't my change save?" mysteries.

The persistence context is a git staging area. new Post() is an untracked file (transient). persist/find = git add (managed, being watched). Flush/commit = the actual commit. Detached = you copied the file out of the repo — edits there change nothing.
Transientnew Post() — no id, unknown to Hibernate and the DB.
ManagedAfter persist() / find() / query — tracked; changes auto-flush.
DetachedContext closed or detach() — has an id, no longer tracked.
Removedremove() — scheduled for DELETE at flush.
States in motion
@Transactional
public void lifecycleTour(Long postId) {
    Post fresh = new Post(…);            // TRANSIENT
    em.persist(fresh);                   // MANAGED — INSERT queued for flush

    Post loaded = em.find(Post.class, postId);   // MANAGED (and now L1-cached)
    loaded.setCaption("golden hour 🌅");         // still just a memory write

    em.detach(loaded);                   // DETACHED — edits below are ignored
    loaded.setCaption("this never persists");

    Post merged = em.merge(loaded);      // copy state onto a MANAGED instance
    em.remove(merged);                   // REMOVED — DELETE queued
}                                        // commit → flush executes the SQL
📸 The classic detached bug

Controller loads a Post in transaction A, stores it in an HTTP session, edits it during the next request, and… nothing saves. The entity went detached when tx A closed. Fix: re-load by id inside the new transaction (or merge), then edit.

Three behaviors of the persistence context that feel like magic until you name them: it remembers what it loaded (L1 cache), compares current state to a load-time snapshot (dirty checking), and writes the diff at flush time.

No save() call, yet it saves
@Transactional
public void editCaption(Long postId, String newCaption) {
    Post post = postRepository.findById(postId).orElseThrow();  // SELECT, snapshot taken
    post.setCaption(newCaption);
    // no repository.save(post) — commit flushes:
    // UPDATE posts SET caption=? WHERE id=?        (dirty checking)
}

@Transactional(readOnly = true)
public void l1CacheDemo(Long postId) {
    Post a = postRepository.findById(postId).orElseThrow();  // SQL SELECT
    Post b = postRepository.findById(postId).orElseThrow();  // NO SQL — L1 cache hit
    assert a == b;      // same Java object: identity guarantee within one context
}

When does flush actually happen?

  • Commit — always.
  • Before a JPQL/native query whose results could be affected by pending changes (so you never read stale data from your own tx).
  • Manually — em.flush() / repo.saveAndFlush(), e.g., to force a constraint violation to surface now.
📸 On Instagram

A moderation job loads a post, tweaks three fields based on rules, and just returns — three field changes become one UPDATE at commit. And within a "hydrate the post screen" transaction, five different helpers calling findById(42) cost one SELECT total.

The L1 cache is per-persistence-context, not global. Two concurrent requests each have their own; it protects you from re-querying yourself, not from seeing other users' committed changes. (The optional shared cache is L2 — a different, opt-in feature.)

EAGER loads the association with the owner, always, whether you use it or not. LAZY gives you a proxy that loads on first access — if the persistence context is still open. Defaults: @ManyToOne/@OneToOne are EAGER (bad), @OneToMany/@ManyToMany are LAZY (good).

House rule with no exceptions: declare everything LAZY (@ManyToOne(fetch = FetchType.LAZY)), then fetch eagerly per query where needed (JOIN FETCH / EntityGraph, ch. 27). EAGER is a global decision you can never undo per-use-case; LAZY is a default you can override anywhere.
📸 The LazyInitializationException, live

Service returns a Post entity; the JSON serializer later touches post.getComments() — but the transaction closed with the service method. Boom: LazyInitializationException. With open-in-view: true it "works" by silently querying during rendering — hiding an N+1 you'll rediscover at 3am.

The failure and the fix
// ❌ throws LazyInitializationException after tx ends
@Transactional(readOnly = true)
public Post broken(Long id) {
    return postRepository.findById(id).orElseThrow();
}   // caller later touches post.getComments() → proxy has no session

// ✅ decide the fetch shape inside the transaction, return a DTO
@Transactional(readOnly = true)
public PostDetailDto detail(Long id) {
    Post post = postRepository.findWithCommentsById(id).orElseThrow(); // ch.27 fetch join
    return PostDetailDto.from(post);          // fully initialized before tx closes
}
EAGER doesn't just over-fetch — on collections it multiplies. Two EAGER @OneToManys fetched with joins produce a cartesian product (MultipleBagFetchException with Lists, or silent row explosion with Sets).

Cascading propagates persistence operations along a relationship: save the parent, children save too. It encodes ownership — use it only where the child's life is fully contained in the parent's.

Post.java — comments live and die with the post
@OneToMany(mappedBy = "post",
           cascade = CascadeType.ALL,
           orphanRemoval = true)
private List<Comment> comments = new ArrayList<>();

// delete a single comment: just unlink it
post.getComments().remove(comment);   // orphanRemoval → DELETE FROM comments …

// delete the post: comments cascade-deleted, no manual cleanup
postRepository.delete(post);

orphanRemoval vs CascadeType.REMOVE: REMOVE fires only when the parent is deleted. orphanRemoval additionally deletes a child the moment it's unlinked from the collection. "Remove comment from post" is exactly that unlink.

📸 The direction matters

Post → Comments: cascade ✓ (comments are meaningless without their post). Comment → User or Post → User: cascade ✗ — imagine CascadeType.REMOVE on Post.user: deleting one photo deletes the account. Cascade flows from owner to owned, never toward shared entities.

Cascade REMOVE deletes row-by-row through Hibernate (N DELETEs + memory). For a post with 50k comments, prefer a bulk @Modifying DELETE FROM Comment c WHERE c.post.id = :id, or let an ON DELETE CASCADE FK do it in the database.
Part V — Advanced

Two answers to concurrent writes. Optimistic: assume conflicts are rare — version-stamp the row, detect collisions at commit, retry the loser. Pessimistic: assume conflicts are likely — take a DB row lock (SELECT … FOR UPDATE) so others wait.

📸 Two moderators, one caption

Both open post #42, both edit, both save. Without locking, the second save silently erases the first (lost update). With @Version, the second save fails loudly and the moderator re-fetches. Meanwhile "reserve this username" during signup is contention-by-design → pessimistic lock.

Optimistic — @Version + retry
@Entity
public class Post {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Version                     // Hibernate manages it; never set it yourself
    private Long version;
    …
}
// UPDATE posts SET caption=?, version=6 WHERE id=? AND version=5
// 0 rows matched → ObjectOptimisticLockingFailureException

@Retryable(retryFor = ObjectOptimisticLockingFailureException.class,
           maxAttempts = 3, backoff = @Backoff(delay = 50))
@Transactional
public void editCaption(Long postId, String caption) {
    Post post = posts.findById(postId).orElseThrow();   // fresh version each attempt
    post.setCaption(caption);
}
Pessimistic — SELECT … FOR UPDATE

Constraints make bad data impossible; indexes make good queries fast. JPA can declare both on the entity — useful as living documentation and for ddl-auto: validate — but the authoritative definition belongs in your Flyway migration.

📸 Why (user_id, created_at) and not two indexes

The profile grid runs WHERE user_id = 7 ORDER BY created_at DESC LIMIT 12. The composite index satisfies filter and sort in one B-tree walk — no filesort. Order matters (leftmost prefix rule): it also serves plain WHERE user_id = ?, but not WHERE created_at > ? alone.

The unique constraint on likes(user_id, post_id) is your correctness layer for double-tap races: two concurrent likes → one INSERT succeeds, one throws DataIntegrityViolationException which you map to a harmless 200/409 (ch. 29). Application-level "check then insert" alone can never guarantee this.
MySQL creates an index for every FK automatically — but doesn't drop your redundant ones. And every extra index taxes each INSERT on a write-heavy table like likes. Index for the queries you actually run, verified with EXPLAIN.

Stop loading entities when you only need columns. Projections make the SELECT fetch exactly the fields a screen needs — less I/O, less memory, no lazy-loading traps, and read-only by construction.

📸 A feed card ≠ a Post entity

The feed shows: post id, image, caption, author's username + avatar, like count. It does not need the full Post + User + Profile object graph with change-tracking snapshots. At 50 cards per scroll × millions of scrolls, projections are the difference.

Open (SpEL) interface projections like @Value("#{target.user.username}") quietly load the whole entity first, then project in memory — you lose the entire benefit. Stick to closed projections (plain getters / constructor expressions) and verify the generated SELECT in the SQL log.
Team rule that pays off: entities for writes, projections for reads. Controllers should never see an @Entity — it also stops accidental lazy-loading during JSON serialization (ch. 20).

Spring Data fills createdAt / updatedAt / createdBy / updatedBy automatically via an entity listener. Define it once in a @MappedSuperclass, extend it everywhere, and stop hand-setting timestamps.

📸 On Instagram

"Edited" labels on captions come from updatedAt > createdAt. When trust-and-safety asks "who changed this post's caption and when?", updatedBy + updatedAt answers without any extra code on any write path.

Need row-level history (every version, not just the last touch)? That's Hibernate Envers (@Audited) or an append-only events table — Spring auditing tracks only the latest state.

Writing 100k rows one INSERT at a time is 100k network round-trips. JDBC batching groups statements per trip; the persistence context just needs periodic draining so it doesn't hoard every entity in memory.

Chunked import with flush + clear
@Transactional
public void importFollowers(Long userId, List<Long> followerIds) {
    User following = users.getReferenceById(userId);
    int i = 0;
    for (Long fid : followerIds) {
        em.persist(new Follow(users.getReferenceById(fid), following));
        if (++i % 50 == 0) {       // align with jdbc.batch_size
            em.flush();            // push the batch over the wire
            em.clear();            // drop managed entities → flat memory
        }
    }
}
📸 Account-migration day

A creator imports 500k followers from another platform. Naive save loop: ~500k round-trips and an OOM as the persistence context accumulates every Follow. Batched + chunked: ~10k round-trips, constant memory, one transaction (or chunk-per-transaction for restartability).

The chapter-04 trade-off bites here: GenerationType.IDENTITY disables Hibernate's INSERT batching (each insert must run to learn its id). Options on MySQL: accept it for normal traffic and drop to JdbcTemplate.batchUpdate for true bulk jobs, or pre-assign ids (app-generated). Also: saveAll() alone is not batching — without jdbc.batch_size it's just a loop.

📸 On Instagram

Feed = posts + author + author's avatar. Rule of thumb: fetch-join the to-one path (user, profile), batch or count-query the to-many path (comments preview, like counts) — and for the hot feed itself, a projection beats all of it.

Fetch join on a collection + Pageable = in-memory pagination. Hibernate logs HHH90003004 firstResult/maxResults specified with collection fetch; applying in memory — it loads everything, then slices in the JVM. Fix with the two-step pattern: page over ids first, then fetch the graph for just those ids (WHERE p.id IN :pageIds). To-one fetch joins paginate fine.
Make N+1 a test failure, not a prod incident: with Hypersistence Utils' SQLStatementCountValidator (or datasource-proxy), assert assertSelectCount(1) on feed-shaped queries in integration tests.
Part VI — Production-grade

Three defensive rings: Bean Validation on request DTOs (reject bad input at the door, per-field errors), optional entity-level annotations (Hibernate validates before flush), and DB constraints as the final, race-proof authority.

📸 Why all three rings

DTO validation gives the app a friendly "username must be 3–30 chars" per field. But two people submitting @purus in the same millisecond both pass existsByUsername — only the unique constraint (ch. 23) settles that race. Validation is UX; constraints are truth.

Validate requests, not entities, as your primary line: by the time an entity fails validation you're mid-flush inside a transaction, and the error is uglier to map. Groups (@Validated(OnCreate.class)) handle create-vs-update rule differences.

One @RestControllerAdvice translates persistence-layer exceptions into stable HTTP semantics. The API contract stays clean even when the cause is three layers down in Hibernate.

GlobalExceptionHandler.java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 400 — validation failures, field-by-field
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public Map<String, String> invalid(MethodArgumentNotValidException ex) {
        return ex.getBindingResult().getFieldErrors().stream()
                 .collect(Collectors.toMap(FieldError::getField,
                                           FieldError::getDefaultMessage,
                                           (a, b) -> a));
    }

    // 404 — our own domain exception
    @ExceptionHandler(PostNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ApiError notFound(PostNotFoundException ex) {
        return new ApiError("POST_NOT_FOUND", ex.getMessage());
    }

    // 409 — unique constraint hit (double-like, taken username)
    @ExceptionHandler(DataIntegrityViolationException.class)
    @ResponseStatus(HttpStatus.CONFLICT)
    public ApiError conflict(DataIntegrityViolationException ex) {
        return new ApiError("CONFLICT", "That action collides with existing data");
    }

    // 409 — optimistic lock lost (ch. 22)
    @ExceptionHandler(ObjectOptimisticLockingFailureException.class)
    @ResponseStatus(HttpStatus.CONFLICT)
    public ApiError staleEdit(ObjectOptimisticLockingFailureException ex) {
        return new ApiError("STALE_EDIT", "Content changed — reload and retry");
    }
}
📸 The double-tap race, resolved politely

Two ❤️ taps race past existsBy…; the second INSERT trips uq_like_user_post → DataIntegrityViolationException → 409 (or treat as idempotent success). No stack trace leaks, no 500, the heart stays red.

Know the translation chain: Hibernate's ConstraintViolationException arrives wrapped as Spring's DataIntegrityViolationException. And repository getReferenceById throws EntityNotFoundException only when the proxy is first touched — possibly far from the call site. Prefer findById().orElseThrow(yourDomainException) at API boundaries.

The whole guide compressed into review-comment form. If a PR violates one of these, you now know the chapter that explains why.

Mapping

  • LAZY everything, fetch per-query with JOIN FETCH / EntityGraph. (ch. 20, 27)
  • Unbounded collections are queries, not fields — never Set<User> likedBy on Post. (ch. 06, 07)
  • Join table needs its own data (timestamps)? Promote it to an entity. (ch. 07)
  • EnumType.STRING, always. Sync helpers on bidirectional links. (ch. 04, 06)
  • Careful with Lombok on entities: @Data's equals/hashCode/toString touch lazy fields → surprise queries, LazyInitializationException, or infinite recursion. Use @Getter @Setter and hand-write identity (stable business key or id-with-null-guard).

Querying

  • Paginate every list endpoint. Feeds use Slice/keyset, not deep OFFSET. (ch. 14)
  • Entities for writes, projections for reads. Controllers never see @Entity. (ch. 24)
  • Log SQL in dev (org.hibernate.SQL: debug) and actually read it — count queries per endpoint. (ch. 02, 27)
  • Existence/count questions use existsBy… / countBy…, never "load and check". (ch. 09)

Transactions & concurrency

  • @Transactional on services; readOnly = true on reads; know the checked-exception rollback trap. (ch. 15)
  • No self-invocation of transactional/locked methods — the proxy can't see it. (ch. 15, 16)
  • Counters → atomic UPDATE; user-edited rows → @Version; must-not-fail claims → pessimistic lock. (ch. 22)
  • Uniqueness under concurrency is a DB constraint, never just a pre-check. (ch. 23, 28)

Operations

  • open-in-view: false. ddl-auto: validate + Flyway in prod. (ch. 02)
  • Keep transactions short: no HTTP/Kafka/S3 calls inside a tx — you're holding a pool connection (and maybe row locks) the whole time.
  • Batch bulk writes; remember IDENTITY's insert-batching cost. (ch. 26)
  • Pool sized to reality (~cores×2 per instance), verified under load — not 100.
